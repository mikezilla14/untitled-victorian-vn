# Prose editor quickstart

You are here to write or revise story text. This page is everything you need.

---

## The one rule

**You work in `main-game/draft/`. Never touch `main-game/prod-game/`.**

`main-game/draft/` is the story workspace. `main-game/prod-game/` is the live game code. Edits to the game code are handled separately after your prose is reviewed and approved.

---

## The files you edit

Story drafts live here:

```
main-game/draft/releases/release-1-mvp/days/
  day100/day100_non_canon.rpy
  day101/day101_non_canon.rpy
  day102/day102_non_canon.rpy
  ...
```

Each `_non_canon.rpy` file is a day's worth of scenes. Inside, comments mark what each block is:

| Tag | Means |
|-----|-------|
| `[BEAT]` | A prose moment — dialogue, action, description |
| `[CHOICE]` | A player decision point |
| `[STATE]` | A game state change (do not edit these) |
| `[ASSET]` | A background or sprite call (do not edit these) |

**Edit `[BEAT]` and `[CHOICE]` text freely. Leave `[STATE]` and `[ASSET]` lines alone.**

---

## Sprite placement is automatic (but you can override it)

Where each character sprite stands on screen is handled for you by an automatic tool, so most `show ... at ...` lines are tagged `# [asset auto]` — **don't hand-tune those, they get regenerated.** The full list of these tags is printed at the top of every day file under `SPRITE DIRECTION`.

If you *do* want a specific staging command kept exactly as you wrote it:

| You want to… | Add this |
|--------------|----------|
| Keep one `show` line exactly as written | append ` # [asset keep]` to that line |
| Hand-direct a whole scene (tool leaves it alone) | put `# [asset lock:scene]` on its own line just before the `scene` line |
| Note who enters/leaves a scene | `# [enter:Cora]` / `# [exit:Missy]` on their own line |

Full reference: [`docs/narrative_workflow.md`](../narrative_workflow.md#sprite-direction--asset-lock-tags).

## What you do not touch

| Path | Why |
|------|-----|
| `main-game/prod-game/` | Live game code — changes here go through a separate promotion process |
| `main-game/canon/` | Locked lore — character facts, historical rules, world bible |
| `scripts/` | Validation tooling |
| `.agents/` | AI role definitions |
| `main-game/pipeline/` | Gate verdicts and review artifacts — generated by the review process |

## How to validate your work

From the repo root in a terminal:

```powershell
py scripts/validate.py --profile narrative --agent writers_room --skip-gate-checks --files "main-game/draft/releases/release-1-mvp/days/day101/day101_non_canon.rpy"
```

Replace the file path with the day you edited. If the command exits with no errors, you are good to open a PR.

If you edited multiple days, separate the paths with commas:

```powershell
py scripts/validate.py --profile narrative --agent writers_room --skip-gate-checks --files "main-game/draft/releases/release-1-mvp/days/day101/day101_non_canon.rpy,main-game/draft/releases/release-1-mvp/days/day102/day102_non_canon.rpy"
```

## How to open a PR

1. Create a branch: `git checkout -b prose/day101-parlour-scene`
2. Stage your changes: `git add main-game/draft/releases/release-1-mvp/days/day101/`
3. Commit: `git commit -m "revise day 101 parlour confrontation"`
4. Push: `git push -u origin prose/day101-parlour-scene`
5. Open a pull request targeting the branch Michael tells you to use. For now, use `main` unless told otherwise.

In the PR description, note what you changed and why (e.g. "softened Cora's line in the parlour scene — felt too aggressive for her characterisation at this point").
