# Technical Documentation: Two-Tiered Character-Specific Suspicion (Option B)

This document details the architectural additions implemented in the Release 1 sandbox (`main-game/draft/releases/release-1-mvp/`) to support the refined character-specific story chains, choice dynamics, and confrontations. These changes are designed to be 100% backwards-compatible with existing scripts, while laying the technical foundation for Release 2's granular choice design.

---

## 1. Two-Tiered Suspicion Architecture

In `classes_non_canon.rpy` (and promoted to production `classes.rpy`), the `PlayerStats` class has been extended to support individual two-tiered suspicion pools for core witnesses (Stern, Vance, Gideon, and Missy).

### Backing State Variables
Inside `PlayerStats.__init__`, the following state attributes are defined:
- **Base Suspicion (`base_susp`)**: Permanent established suspicion level, clamped to `[0, 100]`.
- **Acute Suspicion (`acute_susp`)**: Temporary volatile heat generated by player interactions, clamped dynamically to `[0, 100 - base_susp]`.
- **Tracked Witnesses**: `self.tracked_characters = ["stern", "vance", "gideon", "missy"]`
- **Asymptotic Decay Rates (`decay_rates`)**: Active decay rates applied to acute suspicion during appeasements:
  - Stern: `0.15` (slow decay)
  - Vance: `0.60` (rapid reset)
  - Gideon: `0.50` (shifting equation)
  - Missy: `0.90` (instant reset)

### Enforced Sum Clamping & Headroom Squeezing
To mathematically guarantee that no single character's total suspicion exceeds 100, the sum of `base + acute` is strictly capped at `100` at all times:
1. **Priority Floor**: Permanent base suspicion takes precedence.
2. **Dynamic Squeezing**: If base suspicion is programmatically increased (e.g. at a milestone) and the sum would exceed 100, acute suspicion is automatically **squeezed** (shrunk) to fit exactly within the remaining headroom:
   $$/text{acute/_susp} = /max(0, /min(100 - /text{base/_susp}, /text{acute/_susp}))$$
   This is enforced inside the core `recalculate_anxiety()` and state update cycles.

### Derived Consolidated Anxiety Score
Global `self.anxiety` is calculated using independent probability (the probability of getting caught by at least one witness):
$$/text{Anxiety} = 100 /cdot (1 - /prod_{i} (1 - /frac{/text{Susp}_i}{100}))$$
Where $/text{Susp}_i$ is the total suspicion (`base + acute`) for each active tracked witness. If Anxiety reaches `100`, the narrative immediately routes to the `game_over_dismissed` fail-state.

## 2. Updated Effect Application Utility

The global `apply_effects` utility in `functions_non_canon.rpy` (and promoted to `functions.rpy`) signature has been updated with expanded parameters to route acute and base suspicion modifications cleanly.

### Function Signature
```python
def apply_effects(insp=0, corr=0, susp=0, 
                  stern_susp=0, stern_base=0, 
                  vance_susp=0, vance_base=0, 
                  missy_susp=0, missy_base=0,
                  gideon_susp=0, gideon_base=0):
```

### Routing Logic
- **Acute Adjustments (`*_susp` parameters)**: Standard, user-controlled interactive outcomes (lies, snooping, appeasing) modify `acute_susp` (temporary heat).
- **Base Deltas (`*_base` parameters)**: Calculated narrative milestones or branch resolutions modify `base_susp` (permanent suspicion).
- **Legacy Deprecation**: If general `susp` is called with a non-zero value, it raises a `ValueError` reminding developers that suspicion must target a specific witness.

At the end of effect application, `player.update_stats()` is called to compile modifications, enforce squeezing, and recalculate global `anxiety`.

## 3. Backwards-Compatibility Layer

To protect existing creative scripts and menu branches from breaking, the following compatibility adapters are built into the framework:

1. **Getter Properties**: `@property` getter wrappers are defined for legacy variables:
   - `player.stern_suspicion`
   - `player.vance_suspicion`
   - `player.missy_suspicion`
   - `player.gideon_suspicion`
   
   These properties automatically return the total combined suspicion (`base + acute`) capped at `100`. Existing script checks (e.g. `if player.stern_suspicion >= 50:`) work flawlessly.
2. **Compatibility Method Wrapper**: The deprecated method `adjust_character_suspicion(character, amount)` redirects to `add_suspicion(character, acute_amount=amount)`, preserving compatibility with external scripts or tests.

## 4. Writing & Implementation Guidelines (Option B)

- **Interactive Evasion**: When writing standard choice outcomes, adjust acute suspicion using `apply_effects(stern_susp=15)` to give immediate heat, allowing the player to manage or decay it.
- **Narrative Milestones**: At major day transitions, plot intersections, or calculated branch outcomes, apply permanent base suspicion deltas using `apply_effects(stern_base=stern_base_delta)`.

## 5. Chain Outcome and Abandon API (2026-06-28)

Optional story chains now distinguish **safe abandon** from **charged progress**:

| Helper | Effect |
|--------|--------|
| `story.abandon_chain_beat(character)` | Sets `*_chain_closed = True`, `*_chain_outcome = "abandoned"`, does **not** increment `*_chain_level`. |
| `story.complete_chain_beat(character, path="safe_progress")` | Increments level, sets tier outcome (`cautious` / `pact` / `voyeur` at T1–T2). |
| `story.complete_chain_beat(character, path="climax")` | Increments to tier 3 completion; sets `marked` (Stern), `climax` (Missy), or `collusion` (Vance). |

`story.chain_available(character)` returns `False` when `*_chain_closed` is True or level ≥ `MAX_CHARACTER_CHAIN_LEVEL`.

Public setters: `set_stern_chain_outcome`, `set_missy_chain_outcome`, `set_vance_chain_outcome` (whitelisted strings in `StoryState`).
