# INTEGRATION & SYNTAX VALIDATION GUIDE

This guide details the procedure for importing the external LLM's branching code output back into the visual novel's day scripts, and how to validate the syntax and compliance of the integrated code.

---

## 1. The Import Header Metadata

All code generated by the external LLM must begin with the standard Ren'Py metadata comment block:

```renpy
# =========================================================================
# METADATA: BOOK ENGINE IMPORT HEADER
# Target File: main-game/non-prod-game/game/days/book1_day101_non_canon.rpy
# Target Label: book1_block_day1_alt_predator_core
# Target Chapter: day1_chapter
# Target Archetype: predator
# Style Lens: Sweeney Todd
# Generation Mode: multi_branch_code
# =========================================================================
```

### Parsing Logic (for Agents/Tooling):
1. **Locate Target:** Extract `Target File` and `Target Label` from the header metadata.
2. **Read File:** Open the specified `Target File` (paths must be treated as relative to the workspace root, utilizing forward slashes).
3. **Locate Block:** Search for `label [Target Label]:` (e.g. `label book1_block_day1_alt_predator_core:`).
4. **Identify Block Scope:** The block starts at the label line and extends down to the matching `return` statement (or until the next unindented `label` statement begins).
5. **Replace Block:** Overwrite the old block with the generated code from the LLM, leaving the import header intact at the top of the label as documentation.

---

## 2. Indentation & Formatting Rules

Ren'Py has strict script syntax that the LLM code must respect:

* **Indentation Floor:** The label header is unindented. All commands under the label must be indented by exactly **4 spaces**.
* **Conditional Indentation:** `if`, `elif`, and `else` statements must align with the label contents (4 spaces). All script statements inside conditional blocks (e.g. `call`) must be indented by exactly **8 spaces** (an additional 4 spaces).
* **Quotes and Syntax:** All text strings in `call book1_nvl_write_line("...")` must be wrapped in valid double quotes. If the text itself contains double quotes, they must be escaped using a backslash (`\"`) or wrapped in single quotes.
* **Writer Monologues:** Cora's inner thoughts must be wrapped in `cora_inner "..."` syntax, not standard dialogue calls.

---

## 3. Code Validation & Compliance Checks

Once the code is integrated into the file, you must run the following validation pipeline to ensure it matches project standards and compiles correctly.

### Step 1: Pre-PR Validation Linter
Run the project linter on the modified file to check script structure, label references, and syntax. Set the agent to `writers_room` and skip gate checks since the draft is non-canon:
```powershell
python scripts/validate.py --profile changed --agent writers_room --skip-gate-checks --files "main-game/non-prod-game/game/days/book1_day101_non_canon.rpy"
```

### Step 2: Contract validation
Verify that the story variables and release metrics meet the engine's design contract:
```powershell
python scripts/contract_validate.py --day day101 --release release-1-mvp
```

### Step 3: Historical Anachronism Check
Run the project's historical linter on the modified file to scan the LLM-generated prose for any period-inaccurate vocabulary or modern slang:
```powershell
python scripts/historical_linter.py --files "main-game/non-prod-game/game/days/book1_day101_non_canon.rpy"
```

### Step 4: Staged Sandbox Testing
To manually verify the branching outputs, run the project's dev harness:
1. Open the project in Ren'Py or run the non-prod development build.
2. Select the **Day 1 writing test setup** menu.
3. Choose the target path (e.g. *Run: Corruption Test Run* or *Manual Render*) to trace how the newly imported code renders on the NVL screen.
